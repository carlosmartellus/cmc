# {{PROJECT_NAME}}: CMC Dedicated Framework

## Philosophy
**CMC (Carlos Martel Cornejo)** is a framework built for developers who demand absolute control over their infrastructure. 

### Core Principles:
1. **Scaffolding Automation**: Rapidly forge your project's skeleton without manual boilerplate.
2. **SQL-First (No-ORM)**: CMC does NOT use an ORM. We believe in the power and clarity of raw SQL.
   - **Performance**: No hidden layers between your logic and PostgreSQL.
   - **Optimization**: All `JOIN` operations and index management are manual, putting optimization directly in your hands.
3. **Migration Manager**: CMC uses it's own migration manager, but respecting SQL and developer implementations.
4. **Python 3.13.X**: Powered by the latest Python features for maximum concurrency and efficiency.



---

## Project Structure

- **sv{{PROJECT_NAME}}/** (Server): Your Python 3.13 backend.
  - `/handlers`: Pure SQL CRUD logic generated by the CMC engine.
  - `/migrations`: Your database's genetic history in raw SQL.
- **cl{{PROJECT_NAME}}/** (The Cockpit): Your frontend forged with Vite or Tauri.
- **commons/** (The Core): Shared logic and configuration.
  - `config.toml`: Your infrastructure's brain.
  - `db_manager.py`: The raw PostgreSQL engine.

## Database Philosophy
CMC assumes you know your data better than any library.
- Use `psycopg2` for execution.
- Complex relations are handled via manual SQL queries.
- Scalability is achieved by optimizing the DB engine, not by abstracting it.

## Getting Started

### 1. Initialize the Structure
Forging your project's skeleton is the first step. Open your terminal and run:

* **For a fresh start**:  
    `cmc init --new [project_name]`
* **To adopt an existing database**:  
    `cmc init --existing`

### 2. Choose Your Cockpit
During the initialization, the orchestrator will ask for your preferred interface. This choice defines the directory `cl[ProjectName]`:

* **Web**: Forges a high-performance **Vite + React + TS** environment.
* **App**: Forges a native desktop application using **Tauri**.

### 3. Engine Configuration
The wizard will request infrastructure parameters such as PostgreSQL credentials, ports, and hosts. 

While the setup handles this automatically, you can manually fine-tune your environment (switching between `dev` and `prod`) at any time by editing:
`commons/config.toml`

It also provides a switch script:
* **Development**:`cmc env --dev`
* **Production**: `cmc env --prod` 

And creates two databases:

* `{PROJECT_NAME}_db_dev`
* `{PROJECT_NAME}_db_prod`

If you want to remove your project from your system, it is highly suggested to use

* `cmc remove`

This way will guarantee removing not only the root proyect, but also both db if it is specified

## Database Management

### 1. Migration Scaffolding
Following the **SQL-First** philosophy, CMC does not generate models from code. Instead, you forge your database schema using raw SQL migrations. To initialize a new migration pair, execute:

`cmc db create [migration_name]`

This command generates a file within the **Backend** at `sv{{PROJECT_NAME}}/migrations/`:
* `[timestamp]_[migration_name].sql`


#### The UP Script (`.sql`)
This file is the blueprint for your data evolution. Use it to define your **DDL (Data Definition Language)**: `CREATE TABLE`, `ALTER TABLE`, `CREATE INDEX`, or any other structural changes. 

**Best Practice**: CMC wraps these files in `BEGIN;` and `COMMIT;` blocks by default. Ensure your logic remains within these boundaries to guarantee **atomicity**—if any part of the migration fails, the entire database state remains untouched.

### 2. Migration Execution
Once your scripts are ready, you must synchronize the **Server** with the database engine. CMC requires an explicit environment flag to prevent accidental deployments to the wrong infrastructure.

#### Commands:
* **Development**: `cmc db migrate --dev` (Default)
* **Production**: `cmc db migrate --prod`



#### How it works:
1.  **Environment Force**: CMC ignores the global `current_env` pointer for this command and loads the specific configuration for the chosen flag, prioritizing safety.
2.  **Delta Identification**: The engine compares local files in `sv{{PROJECT_NAME}}/migrations/` against the internal `cmc_migrations` table in the target instance.
3.  **Sequential Deployment**: All pending migrations are executed in strict chronological order, dictated by their unique timestamps.
4.  **Registration**: Upon success, the migration name is persisted in the database to prevent redundant runs.

#### Transactional Integrity
Because CMC promotes **Pure SQL**, every migration is executed within its own transaction. If a script fails, the database will automatically roll back to its previous state (thanks to the default `BEGIN/COMMIT` blocks), preventing "half-baked" schema updates.

---

### 3. Reverting Changes (Rollback)
If a deployment introduces a flaw, CMC allows you to step back in time. Like the migration command, this is environment-specific and positional.

#### Command:
`cmc db rollback [steps] [--dev | --prod]`

* **Example (Dev)**: `cmc db rollback 1 --dev`
* **Example (Prod)**: `cmc db rollback 2 --prod`



#### Mechanics:
* **The Undo Path**: CMC identifies the last `N` applied migrations (positional `steps`, default is 1) and executes a `DROP` for any migration.
* **Cleanup**: Once the rollback is successful, the records are removed from the `cmc_migrations` table, marking them as "pending" for future corrected deployments.

## API

CMC centralizes route management through the **CMC Contract**, a Single Source of Truth (SSOT) that dictates the communication protocol between the frontend and backend. It serves both as a technical enforcer and a definitive developer guide.

A standard contract defines a route's method, required inputs, and expected outputs. This structure ensures that both ends of the stack stay synchronized without guesswork.

### Entities

Forging an `Entity` is the fastest way to bridge your database with the API:

`cmc api register entity [name]`

**How it works:**
1. **Introspection:** CMC inspects the database to extract column names and metadata.
2. **Scaffolding:** It automatically generates a `CMC CONTRACT` and a full CRUD handler based on your database model.
3. **Refinement:** While the forge handles the boilerplate, you must adjust the input/output definitions in the contract to match your specific needs.

#### The CMC Protocol: Strict ID Requirement

To ensure architectural stability and predictable routing, CMC operates under a strict **Convention over Configuration** model regarding entity identity:

* **Mandatory Primary Key:** Every table registered as an Entity **MUST** have a column named exactly `id`.
* **Surrogate Keys Everywhere:** This requirement applies to all tables, including junction tables (`N:M` relationships). Even if the table only serves to link two other entities, a unique `id` is required for CMC to manage its lifecycle.
* **Failure State:** The registration process will fail with a `Protocol Violation` error if the `id` column is missing, protecting the framework from inconsistent relational states.

> [!WARNING]
> **Manual JOIN Control:** CMC does not auto-populate table relationships or JOINs during the scaffolding process. The generated `Read` handler provides a flat projection of the table columns. If relational data is required, the developer MUST manually refine the `SELECT_SQL` and the `CONTRACT` output. This design choice prevents hidden performance bottlenecks and ensures total agency over the data and columns exposed.

**Crucial Rules:**
* **Never delete the `CONTRACT` block**: It is the brain of the handler.
* **Keep it Simple**: Junction tables should include an `id` to be compatible with the framework's internal logic. Handle complex relational logic or non-id-compliant tables through custom **routes** instead of entities.
